# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

See also `AGENTS.md` in this repo — it's the primary agent-facing guide (coding conventions, package list, common commands) and is kept in sync with this file. Read both.

## What this repo is

Personal dotfiles managed with GNU Stow, targeting Arch Linux (primary) and Debian/Ubuntu. Each top-level directory is a Stow package: its internal path structure mirrors `$HOME` (e.g. `bash/.bashrc` → `~/.bashrc`, `nvim/.config/nvim/` → `~/.config/nvim/`), and Stow symlinks it into place.

## Commands

```bash
stow -t $HOME <package>          # install/symlink a package
stow -D -t $HOME <package>       # remove a package's symlinks
stow -R -t $HOME <pkg1> <pkg2>   # restow (recreate/update) after edits
```

Run from the repo root; `-t $HOME` can be omitted if already `cd`'d into the repo (Stow defaults target to `..` relative to the package dir otherwise, so prefer being explicit).

## Architecture notes

- **`skills/` is special**: it stows *hidden* directories (`.claude/`, `.opencode/`, `.agents/`) into `$HOME`, exposing the same skill set to both Claude Code and OpenCode. The actual skill source lives under `skills/.agent_skills/<skill-name>/`. The paths `skills/.claude/skills/`, `skills/.opencode/skills/`, and `skills/.agents/skills/` are themselves symlinks/generated by Stow pointing back at `.agent_skills` — **never edit files under those three directories directly**; edit the source in `.agent_skills/` or changes can be lost or become inconsistent across tools.
- **`utils/.config/randomcode/`** is a self-contained Python project (uv-managed, has its own `pyproject.toml`/`uv.lock`/`.venv`) nested inside a Stow package — treat it as an independent Python package when working inside it, not as a dotfile.
- Other packages (`bash`, `bat`, `ghostty`, `starship`, `tmux`, `yazi`) are plain config-file trees; `utils` also carries standalone executable scripts (`ns`, `fzftunes`, `mp4thumb`, `tms`, `addskill`) with no shared build system between them.

## Coding conventions (shell scripts)

- `set -Eeuo pipefail` and `shopt -s inherit_errexit`
- No inline comments — use `: "docstring"` as the first line of each function instead
- Logging/status output → stderr; actual data output → stdout
- No `eval`; build commands as arrays
- Parallelism via GNU `parallel`, not `xargs`
- Target is Wayland/KDE — avoid X11-specific tools; use `xdg-open` for URI dispatch
